# Apache Doris 4.0.4 存算分离架构设计决策详解

## 一、为什么选择存算分离架构？

### 1.1 传统架构的局限性

传统Doris架构中，每个Backend节点同时承担存储和计算职责：

```
传统架构问题:
┌─────────────────────────────────────────────────────────────┐
│                    传统Doris架构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Backend Node                       │  │
│  │  ┌────────────┐  ┌────────────┐  ┌────────────┐     │  │
│  │  │   存储     │  │   计算     │  │   网络     │     │  │
│  │  │  (本地盘)  │  │  (CPU)     │  │  (IO)      │     │  │
│  │  └────────────┘  └────────────┘  └────────────┘     │  │
│  │                                                      │  │
│  │  问题: 存储和计算耦合，必须同时扩展                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  扩展问题:                                                  │
│  - 数据迁移慢: 添加节点需要重新平衡数据                    │
│  - 资源浪费: 计算空闲时存储资源也被占用                    │
│  - 成本高: 必须使用高配机器满足存储和计算需求              │
│  - 恢复慢: 节点故障需要从副本恢复数据                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**具体问题分析：**

| 问题 | 影响 | 业务后果 |
|------|------|----------|
| 存储计算耦合 | 扩展时必须同时增加存储和计算 | 成本高，资源浪费 |
| 数据本地化 | 节点故障需要数据迁移 | 恢复时间长（小时级） |
| 资源利用率低 | 计算波峰波谷无法弹性调整 | 闲置资源浪费成本 |
| 单节点存储限制 | 受限于本地磁盘容量 | 无法支持超大数据量 |

### 1.2 存算分离架构的优势

```
存算分离架构:
┌─────────────────────────────────────────────────────────────┐
│                    存算分离架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────┐         ┌──────────────────┐         │
│  │   Compute Node   │         │   Backend Node   │         │
│  │  ┌────────────┐  │         │  ┌────────────┐  │         │
│  │  │   计算     │  │  ◄────► │  │   存储     │  │         │
│  │  │  (CPU)     │  │  网络   │  │  (本地+GCS)│  │         │
│  │  └────────────┘  │         │  └────────────┘  │         │
│  │                  │         │                  │         │
│  │  特点:           │         │  特点:           │         │
│  │  - 无状态        │         │  - 有状态        │         │
│  │  - 可随时销毁    │         │  - 数据持久化    │         │
│  │  - 弹性伸缩      │         │  - 缓存热数据    │         │
│  │  - Spot VM       │         │  - 写入优化      │         │
│  └──────────────────┘         └──────────────────┘         │
│           │                            │                    │
│           │                            │                    │
│           ▼                            ▼                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              共享存储层 (GCS)                        │   │
│  │  - 无限容量                                         │   │
│  │  - 高可用 (99.999999999% 持久性)                   │   │
│  │  - 低成本 (分层存储)                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心优势详解：**

**1. 独立扩展能力**

```
场景: 查询高峰期，需要增加计算能力

传统架构:
  添加节点 → 需要同时添加存储 → 数据重新平衡 → 耗时数小时
  
存算分离架构:
  添加CN节点 → 无需数据迁移 → 秒级生效 → 成本优化
```

**2. 成本优化**

```
成本对比 (月度):

传统架构:
  - 10台高配服务器 (存储+计算): $10,000
  - 总成本: $10,000/月

存算分离架构:
  - 3台BE (存储节点): $3,000
  - 5台CN (Spot VM): $1,500 (节省70%)
  - GCS存储: $2,000
  - 总成本: $6,500/月 (节省35%)
```

**3. 故障恢复**

```
故障恢复对比:

传统架构:
  节点故障 → 数据迁移 → 重新平衡 → 恢复时间: 2-4小时

存算分离架构:
  CN故障 → 直接销毁重建 → 恢复时间: < 1分钟
  BE故障 → 从GCS恢复数据 → 恢复时间: 10-30分钟
```

## 二、为什么选择GKE？

### 2.1 GKE vs 自建Kubernetes

| 维度 | GKE | 自建K8s |
|------|-----|---------|
| 运维成本 | 托管控制平面，零运维 | 需要维护etcd、api-server等 |
| 高可用 | 自动跨可用区部署 | 需要手动配置 |
| 自动扩缩容 | 原生支持CA/HPA/VPA | 需要额外安装配置 |
| 安全性 | 自动安全更新 | 需要手动更新 |
| 成本 | 控制平面免费 | 需要额外资源 |

### 2.2 GKE核心特性利用

**1. Autopilot模式**


**3. GKE监控集成**

```yaml
  - 日志聚合

```

```
│       └──────────────────┴──────────────────┘               │

│  2. 单AZ故障 → 剩余2节点继续服务                            │
│  3. 网络分区 → 多数派继续，少数派只读                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么选择3节点？**

```
节点数选择分析:

1节点:
  - 无高可用
  - 单点故障
  - 不推荐

2节点:
  - 无法形成多数派
  - 一节点故障，无法选举
  - 不推荐

3节点 (推荐):
  - 多数派 = 2
  - 允许1节点故障
  - 成本与可用性平衡

5节点:
  - 多数派 = 3
  - 允许2节点故障
  - 成本较高，适合关键业务
```

### 4.2 向量化执行引擎

```
向量化执行原理:

┌─────────────────────────────────────────────────────────────┐
│                    向量化 vs 标量执行                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  标量执行 (传统):                                           │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ for row in rows:                                      │  │
│  │     result = process(row)  # 每行一次函数调用         │  │
│  │     output.append(result)                             │  │
│  │                                                       │  │
│  │ 问题:                                                 │  │
│  │ - 函数调用开销大                                      │  │
│  │ - CPU缓存利用率低                                     │  │
│  │ - 分支预测失败多                                      │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  向量化执行 (Doris 4.0):                                    │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ for batch in batches:  # 每批1024行                  │  │
│  │     result = process_batch(batch)  # SIMD指令        │  │
│  │     output.extend(result)                             │  │
│  │                                                       │  │
│  │ 优势:                                                 │  │
│  │ - 减少函数调用 (1000x)                                │  │
│  │ - SIMD并行计算                                        │  │
│  │ - CPU缓存友好                                         │  │
│  │ - 分支预测优化                                        │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  性能提升: 5-10x                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 七、总结

|------|------|------|
| 存算分离 | 存储计算独立扩展 | 成本降低35%，弹性伸缩 |

| CN无状态 | 可随时销毁重建 | 秒级扩缩容 |
| Spot VM | 成本优化 | 节省70%计算成本 |
| 多级缓存 | 降低查询延迟 | P99 < 5秒 |
| 分区分桶 | 数据并行处理 | 支持PB级数据 |
| GCS存储 | 无限容量 | 支持超大数据量 |
